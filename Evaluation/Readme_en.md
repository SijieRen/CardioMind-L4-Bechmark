# CardioBench Evaluation Code Usage and Implementation Documentation

## 1. Document Overview

This document details the core functions, implementation logic, scoring system and usage methods of the CardioBench evaluation code. CardioBench is the first Level-4 clinical benchmark for cardiovascular specialties. Its evaluation code can systematically quantify the diagnostic reasoning ability, evidence traceability ability and safety (hallucination rate) of large language models in real clinical scenarios.



## 2. Core Inputs and Outputs

### 2.1 Input Data

The evaluation code accepts the following three types of standardized inputs:

1. **Predicted Diagnosis Report**: A complete cardiovascular disease diagnosis report generated by the LLM to be evaluated, which must include primary diagnosis, secondary diagnosis, differential diagnosis and corresponding evidence
2. **Gold Standard Labels**: Structured diagnostic results annotated by cardiology experts, including ICD-10 standardized disease nomenclature, evidence chains and recommended examinations
3. **Complete Electronic Medical Record (EMR)**: The patient's original clinical records, including chief complaint, present illness history, past medical history, laboratory tests and imaging results

### 2.2 Output Results

The evaluation code generates a multi-dimensional quantitative report, including:

- Overall weighted clinical score
- Sub-scores for primary/secondary/differential diagnosis
- Evidence extraction precision, recall and F1 score
- Hallucination rate statistics for various types of evidence
- Subset-specific performance across clinical subsets (Mild/Typical/Critical/Rare)
- Model capability radar chart and comparative analysis



## 3. Core Implementation Logic

### 3.1 LLM-assisted Evaluator Design

- Uses large language models as auxiliary evaluators to perform evaluation tasks through carefully designed text prompts
- A prompt suite optimized through multiple iterations, minimizing API calls while ensuring the reliability of clinical judgments
- Built-in ICD-10 standardized disease nomenclature system, supporting matching and verification of diagnostic expressions at different granularities
- Supports semantic understanding and evidence alignment of Chinese clinical texts

### 3.2 Diagnostic Accuracy Evaluation

#### 3.2.1 Primary Diagnosis Evaluation

- **Top-1 Accuracy**: The proportion of predicted primary diagnoses that exactly match the gold standard
- **Top-k Accuracy**: The proportion of gold standard primary diagnoses that appear in the model's non-excludable differential candidate list (a Top-1 hit automatically counts as a Top-k hit)
- **Etiology Classification Score**: Evaluates the model's ability to identify disease phenotypes/molecular subtypes

#### 3.2.2 Secondary and Differential Diagnosis Evaluation

- Uses the **Intersection over Union (IoU)** metric to evaluate both the accuracy and completeness of the diagnosis set
- Balances the penalties for missed diagnoses (failing to list existing diseases) and misdiagnoses (listing non-existent diseases)
- Differential diagnosis additionally evaluates **Exclusion Precision**: The model's ability to correctly exclude diseases

#### 3.2.3 Diagnostic Term Matching Rules

- If the predicted diagnosis is a hypernym (broader term) of the gold standard, it is directly determined to be inconsistent
- If the predicted diagnosis is a hyponym (more specific term) of the gold standard, its clinical validity needs to be evaluated in conjunction with the patient's medical record
- Supports automatic mapping of synonyms and common clinical abbreviations

### 3.3 Evidence Extraction and F1 Calculation

- Adopts a **comma-separated phrase-level evaluation system**, taking each segment separated by Chinese commas and periods as the smallest evaluation unit
- Calculates precision, recall and F1 score separately for all evidence fields:
  - Primary diagnosis evidence
  - Secondary diagnosis evidence
  - Differential diagnosis supporting evidence (positive evidence)
  - Differential diagnosis exclusion evidence (negative evidence)
- Recommended examinations (Check) are evaluated separately for their appropriateness, but are not included in hallucination evaluation (due to the temporal dynamics of clinical examinations making hallucination boundaries ambiguous)

### 3.4 Hallucination Detection Mechanism

- **Hallucination Definition**: Generated evidence statements that cannot be verified in the patient's written medical record
- **Detection Scope**: Covers all evidence fields and independently quantifies different types of hallucinations
- Judgment rules:
  1. The evaluated statement does not appear in the patient's medical record â†’ Judged as a hallucination
  2. The evaluated statement contains a negation (e.g., "no fever"), but the negated content (e.g., "fever") actually exists in the medical record â†’ Judged as a hallucination
- During detection, the predicted evidence is compared with both the reference labels and the complete medical record to ensure accuracy

### 3.5 Majority Voting Mechanism

- To reduce random bias, a majority voting mechanism is adopted
- Each evaluation item is subjected to a maximum of 9 independent evaluations
- When a majority consensus of **â‰¥5 identical judgments** is reached, the loop is immediately terminated and the result is adopted



## 4. Scoring Calculation System

All scoring weights are set based on clinical importance, reflecting the impact of different diagnostic tasks on patient prognosis.

### 4.1 Overall Clinical Score

```
Overall = 0.5 Ã— PD_Total + 0.3 Ã— SD_Total + 0.2 Ã— DD_Total
```

- Primary Diagnosis (PD) has the highest weight (0.5) as it directly determines treatment plans and prognosis
- Secondary Diagnosis (SD) has a weight of 0.3, reflecting the impact of comorbidities on drug selection, contraindications and overall management
- Differential Diagnosis (DD) has a weight of 0.2, mainly serving as a diagnostic safety net to prevent missed diagnoses

### 4.2 Primary Diagnosis Total Score

```
PD_Total = 0.4Ã—Top-1 Prec. + 0.2Ã—Top-k Prec. + 0.2Ã—Etiol. Score + 0.2Ã—Evidence F1
```

- Top-1 Accuracy has the highest weight (0.4), reflecting the critical importance of correct initial diagnosis
- Top-k Accuracy (0.2) serves as a safety net metric; even if the model does not rank the correct diagnosis first, it still has clinical value as long as it is included in the candidate list
- Etiology Classification (0.2) and Evidence F1 (0.2) have the same weight, emphasizing the specificity and traceability of diagnosis

### 4.3 Secondary Diagnosis Total Score

```
SD_Total = 0.67Ã—IoU + 0.33Ã—Evidence F1
```

- IoU has the dominant weight (0.67), emphasizing that the comorbidity list needs to be "clean and complete", neither missing important comorbidities nor over-diagnosing
- Evidence F1 (0.33) ensures that each claimed comorbidity is supported by medical record documentation

### 4.4 Differential Diagnosis Total Score

```
DD_Total = 0.4Ã—IoU + 0.3Ã—Excl. Prec. + 0.1Ã—Pos. Evidence F1 + 0.1Ã—Neg. Evidence F1 + 0.1Ã—Check F1
```

- IoU (0.4) evaluates the completeness and conciseness of the differential diagnosis list
- Exclusion Precision (0.3) has a relatively high weight, reflecting the clinical value of correctly excluding diseases (avoiding unnecessary examinations and treatments)
- Supporting evidence and exclusion evidence F1 each account for 0.1, requiring the model to provide bidirectional evidence support
- Recommended examination F1 (0.1) evaluates the model's ability to propose the next diagnostic steps

### 4.5 Weighted Hallucination Score

```
Hall_weighted = 0.5Ã—PD_hall + 0.3Ã—SD_hall + 0.1Ã—DD_pos_hall + 0.1Ã—DD_neg_hall
```

- The weights are consistent with the overall score, and hallucinations in primary diagnosis evidence carry the highest safety risk
- A lower score indicates better evidence fidelity and higher clinical safety

### 4.6 Cross-subset Weighted Average

When calculating overall performance, the sample size of each clinical subset is used as the weight to ensure that the indicators reflect the incidence rate in real clinical scenarios:

- Mild: 397 cases (outpatient, incomplete information)
- Typical: 163 cases (inpatient, complete examinations)
- Critical: 147 cases (life-threatening, requiring emergency treatment)
- Rare: 101 cases (prevalence < 1/10000)



## 5. Evaluation Process

1. **Data Preparation**: Organize the diagnosis reports of the model to be evaluated, expert-annotated gold standards and original electronic medical records into the specified JSON format

   Prepare an input file in **jsonl format**, with one case per line, which must contain the following 4 fields:

   ```
   {
     "seq_id": "Unique case ID",
     "raw_query": "Complete electronic medical record text",
     "ans_label": "Expert-annotated gold standard diagnosis JSON",
     "response": "Diagnosis report JSON generated by the model to be evaluated"
   }
   ```

2. **Configure Evaluation Parameters**: Set evaluation mode, evaluation rounds, output path, etc.

   |        Parameter        |                         Description                          |        Default Value        |
   | :---------------------: | :----------------------------------------------------------: | :-------------------------: |
   |        `--input`        |                    Input JSONL file path                     | `example_input_cotv3.jsonl` |
   |       `--output`        |                   Output result file path                    |    `results_cotv3.jsonl`    |
   |      `--eval_mode`      |                       Evaluation mode                        |          PD_SD_DD           |
   |      `--evidence`       |         Enable evidence extraction and F1 evaluation         |             Off             |
   |      `--selection`      | Enable differential diagnosis exclusion precision evaluation |             Off             |
   |      `--detailed`       |             Generate detailed evaluation report              |             Off             |
   |       `--format`        |              Output format: `json`/`csv`/`text`              |           `json`            |
   |    `--case_parallel`    |            Enable case-level parallel processing             |             Off             |
   | `--case_parallel_count` |              Number of parallel worker threads               |              3              |
   |  `--pipeline_parallel`  |     Enable pipeline parallelism within evaluation steps      |             Off             |
   |    `--sample_count`     |       Number of evaluation rounds for majority voting        |              9              |

3. **Run Evaluation Script**: Execute the main evaluation program to automatically complete evaluation calculations for all dimensions

   Process the JSONL file and perform basic evaluation

   ```
   /usr/bin/python3 main.py --input example_input_cotv3.jsonl --output results_cotv3.jsonl --eval_mode "PD_SD_DD" --evidence --selection --case_parallel --pipeline_parallel
   
   /usr/bin/python3 main.py --input example_input_cotv3.jsonl --output results_cotv3.jsonl --eval_mode "PD_SD_DD" --evidence --selection --case_parallel --pipeline_parallel
   ```

   Process all diagnosis types and evidence evaluation

   ```
   python main.py --input data.jsonl --output results.json --all --evidence
   ```

   Generate a detailed text report

   ```
   python main.py --input data.jsonl --output results.txt --format text --detailed
   ```

4. **Generate Report**: Output structured evaluation result files

5. **Result Analysis**: Analyze the strengths and weaknesses of the model based on the evaluation report
