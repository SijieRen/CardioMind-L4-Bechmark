# CardioBench 评估代码使用与实现说明文档

## 1. 文档概述

本文档详细介绍了 CardioBench评估代码的核心功能、实现逻辑、评分体系与使用方法。CardioBench 是首个针对心血管专科的 Level-4 临床基准，其评估代码能够系统量化大语言模型在真实临床场景下的诊断推理能力、证据溯源能力与安全性（幻觉率）。



## 2. 核心输入与输出

### 2.1 输入数据

评估代码接收以下三类标准化输入：

1. **预测诊断报告**：待评估 LLM 生成的完整心血管疾病诊断报告，需包含初级诊断、次级诊断、鉴别诊断及对应证据
2. **金标准标签**：由心内科专家标注的结构化诊断结果，包含 ICD-10 标准化疾病命名、证据链与推荐检查
3. **完整电子病历 (EMR)**：患者的原始临床记录，包括主诉、现病史、既往史、实验室检查与影像学结果

### 2.2 输出结果

评估代码生成多维度量化报告，包含：

- 总体加权临床得分
- 初级 / 次级 / 鉴别诊断各分项得分
- 证据提取精确率、召回率与 F1 分数
- 各类型证据的幻觉率统计
- 跨临床子集（轻度 / 典型 / 危重 / 罕见）的细分性能
- 模型能力雷达图与对比分析



## 3. 核心实现逻辑

### 3.1 LLM 辅助评估器设计

- 采用大语言模型作为辅助评估器，通过精心设计的文本提示执行评估任务
- 经过多轮迭代优化的提示词套件，在保证临床判断可靠性的同时最小化 API 调用次数
- 内置 ICD-10 标准化疾病命名体系，支持不同粒度诊断表述的匹配与校验
- 支持中文临床文本的语义理解与证据对齐

### 3.2 诊断准确性评估

#### 3.2.1 初级诊断评估

- **Top-1 准确率**：预测的初级诊断与金标准完全匹配的比例
- **Top-k 准确率**：金标准初级诊断出现在模型不可排除鉴别候选列表中的比例（Top-1 命中自动计为 Top-k 命中）
- **病因分类得分**：评估模型对疾病表型 / 分子亚型的识别能力

#### 3.2.2 次级与鉴别诊断评估

- 采用 ** 交并比 (IoU)** 指标，同时评估诊断集合的准确性和完整性
- 平衡漏诊（未列出真实存在的疾病）和误诊（列出不存在的疾病）的惩罚
- 鉴别诊断额外评估**排除准确率**：模型正确排除疾病的能力

#### 3.2.3 诊断术语匹配规则

- 若预测诊断是金标准的上位词（更宽泛术语），直接判定为不一致
- 若预测诊断是金标准的下位词（更具体术语），需结合患者病历评估其临床有效性
- 支持同义词与常用临床缩写的自动映射

### 3.3 证据提取与 F1 计算

- 采用**逗号分隔的短语级评估系统**，将中文逗号和句号分隔的每个片段作为最小评估单元
- 对所有证据字段分别计算精确率、召回率与 F1 分数：
  - 初级诊断证据
  - 次级诊断证据
  - 鉴别诊断支持证据（阳性证据）
  - 鉴别诊断排除证据（阴性证据）
- 推荐检查（Check）单独评估其适当性，但不纳入幻觉评估（因临床检查的时间动态性使幻觉边界模糊）

### 3.4 幻觉检测机制

- **幻觉定义**：生成的证据陈述无法在患者的书面病历中得到证实
- **检测范围**：覆盖所有证据字段，独立量化不同类型的幻觉
- 判定规则：
  1. 被评估的陈述未出现在患者病历中 → 判定为幻觉
  2. 被评估的陈述包含否定词（如 "无发热"），但被否定的内容（如 "发热"）实际存在于病历中 → 判定为幻觉
- 检测时同时对比预测证据与参考标签和完整病历，确保准确性

### 3.5 多数投票机制

- 为减轻随机偏差，采用多数投票机制
- 每个评估项目最多执行 9 次独立评估
- 当达到 **≥5 个相同判断 ** 的多数共识时，立即终止循环并采用该结果



## 4. 评分计算体系

所有评分权重均基于临床重要性设定，反映不同诊断任务对患者预后的影响程度。

### 4.1 总体临床得分

```
Overall = 0.5 × PD_Total + 0.3 × SD_Total + 0.2 × DD_Total
```

- 初级诊断 (PD) 权重最高 (0.5)，因其直接决定治疗方案与预后
- 次级诊断 (SD) 权重 0.3，反映合并症对药物选择、禁忌症与整体管理的影响
- 鉴别诊断 (DD) 权重 0.2，主要作为诊断安全网，防止漏诊

### 4.2 初级诊断总分

```
PD_Total = 0.4×Top-1 Prec. + 0.2×Top-k Prec. + 0.2×Etiol. Score + 0.2×Evidence F1
```

- Top-1 准确率权重最高 (0.4)，反映首次诊断正确的关键重要性
- Top-k 准确率 (0.2) 作为安全网指标，即使模型未将正确诊断排在首位，只要包含在候选列表中仍有临床价值
- 病因分类 (0.2) 与证据 F1 (0.2) 权重相同，强调诊断的特异性与可追溯性

### 4.3 次级诊断总分

```
SD_Total = 0.67×IoU + 0.33×Evidence F1
```

- IoU 占主导权重 (0.67)，强调合并症列表需 "干净且完整"，既不遗漏重要合并症，也不过度诊断
- 证据 F1 (0.33) 确保每个声称的合并症都有病历记录支持

### 4.4 鉴别诊断总分

```
DD_Total = 0.4×IoU + 0.3×Excl. Prec. + 0.1×Pos. Evidence F1 + 0.1×Neg. Evidence F1 + 0.1×Check F1
```

- IoU (0.4) 评估鉴别诊断列表的完整性与简洁性
- 排除准确率 (0.3) 权重较高，反映正确排除疾病的临床价值（避免不必要的检查与治疗）
- 支持证据与排除证据 F1 各占 0.1，要求模型提供双向证据支持
- 推荐检查 F1 (0.1) 评估模型提出下一步诊断步骤的能力

### 4.5 加权幻觉分数

```
Hall_weighted = 0.5×PD_hall + 0.3×SD_hall + 0.1×DD_pos_hall + 0.1×DD_neg_hall
```

- 权重与总体得分一致，初级诊断证据幻觉的安全风险最高
- 分数越低表示证据保真度越好，临床安全性越高

### 4.6 跨子集加权平均

计算整体性能时，使用各临床子集的样本量作为权重，确保指标反映真实临床场景的发生率：

- 轻度 (Mild)：397 例（门诊，信息不完整）
- 典型 (Typical)：163 例（住院，检查完整）
- 危重 (Critical)：147 例（危及生命，需紧急处理）
- 罕见 (Rare)：101 例（患病率 < 1/10000）



## 5. 评估流程

1. **数据准备**：将待评估模型的诊断报告、专家标注金标准与原始电子病历整理为指定 JSON 格式

   准备**jsonl格式**的输入文件，每行一个病例，必须包含以下 4 个字段：

   ```
   {
     "seq_id": "唯一病例ID",
     "raw_query": "完整电子病历文本",
     "ans_label": "专家标注的金标准诊断JSON",
     "response": "待评估模型生成的诊断报告JSON"
   }
   ```

2. **配置评估参数**：设置 评估模式、评估轮次与输出路径等等

   | 参数                    | 说明                          | 默认值                      |
   | ----------------------- | ----------------------------- | --------------------------- |
   | `--input`               | 输入 JSONL 文件路径           | `example_input_cotv3.jsonl` |
   | `--output`              | 输出结果文件路径              | `results_cotv3.jsonl`       |
   | `--eval_mode`           | 评估模式                      | PD_SD_DD                    |
   | `--evidence`            | 开启证据提取与 F1 评估        | 关闭                        |
   | `--selection`           | 开启鉴别诊断排除准确率评估    | 关闭                        |
   | `--detailed`            | 生成详细评估报告              | 关闭                        |
   | `--format`              | 输出格式：`json`/`csv`/`text` | `json`                      |
   | `--case_parallel`       | 开启病例级并行处理            | 关闭                        |
   | `--case_parallel_count` | 并行工作线程数                | 3                           |
   | `--pipeline_parallel`   | 开启评估步骤内流水线并行      | 关闭                        |
   | `--sample_count`        | 多数投票的评估轮次            | 9                           |

3. **运行评估脚本**：执行主评估程序，自动完成所有维度的评估计算

   对 JSONL 文件进行处理并进行基本评估

   ```
   /usr/bin/python3 main.py --input example_input_cotv3.jsonl --output results_cotv3.jsonl --eval_mode "PD_SD_DD" --evidence --selection --case_parallel --pipeline_parallel
   
   /usr/bin/python3 main.py --input example_input_cotv3.jsonl --output results_cotv3.jsonl --eval_mode "PD_SD_DD" --evidence --selection --case_parallel --pipeline_parallel
   ```

   对所有诊断类型及证据评估进行处理

   ```
   python main.py --input data.jsonl --output results.json --all --evidence
   ```

   生成详细的文字报告

   ```
   python main.py --input data.jsonl --output results.txt --format text --detailed
   ```

4. **生成报告**：输出结构化的评估结果文件
